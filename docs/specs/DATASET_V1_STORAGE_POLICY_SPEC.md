# Dataset V1 Storage Policy Specification

Status: ACCEPTED
Last updated: 2026-06-17

## Purpose

Define concrete storage defaults for production `mirante4d-v1` datasets beyond the tiny bootstrap schema.

`NATIVE_DATASET_V1_SCHEMA_SPEC.md` defines the first minimal schema. This spec defines the intended storage policy as datasets grow into real microscopy workloads.

## Scope

This spec covers:

- chunk and shard sizing
- compression defaults
- multiscale generation
- histograms and per-brick acceleration metadata
- optional no-data validity metadata for dense intensity layers
- checksums and validation modes
- OME-Zarr metadata mirroring policy

## Non-Goals

- Supporting legacy `llsm_viewer` preprocessed datasets.
- Accepting arbitrary OME-Zarr datasets without the Mirante4D manifest.
- Hiding missing acceleration metadata behind runtime fallbacks.

## Dense Intensity Brick Policy

Dense intensity arrays use axes `t, z, y, x`. Channels remain separate layers, not an array axis.

Initial production logical brick defaults:

- 3D intensity: `t=1, z=64, y=64, x=64`.
- 2D intensity: `t=1, z=1, y=256, x=256`.
- If a dimension is smaller than the default brick dimension, use the full
  dimension.
- Time is never grouped across multiple timepoints by default.

Target uncompressed logical brick size:

- preferred: `0.5 MiB` to `2 MiB`
- soft maximum: `8 MiB`

Rationale: logical brick reads must be small enough for interactive
cancellation and cache eviction, but large enough to avoid pathological
metadata overhead. A `64^3` `uint16` brick is about `0.5 MiB`, which is a
reasonable first native-rendering unit.

Future preprocessing may adjust brick shapes to approximate cubic physical
extent for strongly anisotropic data, but any adjustment must be recorded
explicitly in the manifest.

## Sharding Policy

Accepted hard-cutover update:

- Production dense native arrays must use Zarr v3 sharding.
- The logical render/read brick is the Zarr sharding subchunk.
- The Zarr outer chunk is the shard object.
- Old production dense arrays that store one object per logical render brick are
  invalid under the current hard-cutover contract.
- Tiny unsharded arrays may only remain as explicitly scoped format test
  fixtures, not as production import output or representative performance
  evidence.

Default shard policy:

- target uncompressed shard size: `64 MiB`
- hard cap: `256 MiB`
- default spatial grouping: up to `4 x 4 x 4` chunks per shard
- no shard spans multiple timepoints by default

Rationale: sharding reduces file-count overhead while preserving random access at chunk granularity inside the shard.

## Compression Policy

Generated native datasets use the production compression profile by default, including small fixtures, so the normal developer/test path exercises compressed chunk decode and encoded-payload validation.

Production dense intensity defaults:

- codec: Zarr v3 bytes plus `zstd`
- level: `3`
- lossy quantization: forbidden by default
- byte shuffling: off by default until benchmarks prove it improves real workloads through the chosen `zarrs` feature set

Uncompressed storage remains allowed for benchmarks, tiny fixtures, or cases where compression measurably hurts throughput.

Every compression choice must be recorded per array. The reader must reject unsupported codecs clearly.

Current implementation status:

- dense `uint8`, `uint16`, and `float32` production writers emit Zarr v3
  sharded arrays whose inner codec chain uses `bytes` followed by
  `zstd(level=3, checksum=false)`
- per-shard `payload_bytes` and `payload_checksum` are measured over the
  encoded compressed shard payload
- validation accepts the documented default codec chain and explicit uncompressed `bytes` arrays
- validation rejects unsupported codec chains or unsupported Zstd configurations before chunk decode
- full validation verifies shard-level encoded payload checksums through the
  Zarr store layer

## Multiscale Policy

Scale `s0` is always the source-like full-resolution grid.

Additional intensity scales should be generated for datasets with any spatial dimension greater than `256` voxels.

Rules:

- downsample spatial axes only; do not downsample time
- each additional scale halves each spatial dimension greater than `1`
- stop when all spatial dimensions are `<= 64` or the scale has `<= 262144` voxels per timepoint
- each scale records its shape, transform, reduction method, and source scale
- scale transforms must preserve physical/world coordinate correctness
- scale transforms must be center-aligned with the source scale: each output voxel center maps to the physical/world center of its contributing source voxel block
- validation and writing must reject stale origin-only downsample transforms, including packages generated by older local code during greenfield development

Default dense intensity reduction:

- `mean` over the source voxels contributing to each output voxel
- integer outputs round to nearest and clamp to the stored dtype range
- floating outputs remain `float32`

If a layer declares a no-data sentinel policy, reduction must follow `NO_DATA_MASK_POLICY_SPEC.md`: render-invalid source samples are ignored, lower-resolution samples with no valid support become invalid, and the one-voxel invalid dilation policy is applied at each generated scale.

Rationale: multiscales are render and navigation acceleration products, not final quantitative analysis substitutes. Exact analysis uses the appropriate source scale.

## Acceleration Metadata

Renderable dense intensity scales must include:

- per-brick occupancy
- per-brick min
- per-brick max
- per-layer per-scale min/max
- per-layer per-scale histogram
- per-layer per-scale robust percentiles
- range hierarchy for valid-data presence and source-value min/max

Histogram defaults:

- `uint8`: `256` bins
- current `uint16` v1 implementation: `256` bins over the full `0..65535` range
- future higher-resolution `uint16` histograms may be added only with a schema update
- `float32`: `4096` bins over the finite value range

Current implementation status:

- strict native `float32` writer and data-engine exact readback are implemented
- non-finite stored `float32` values are rejected before writing
- TIFF import preserves grayscale IEEE `float32` sources as native `float32` datasets without quantization
- CPU renderer `float32` camera modes, exact pick readout, Z-MIP, and explicit display-window transfer compositing are implemented
- CPU `float32` analysis summaries and ROI box measurements are implemented with exact source values and `f64` accumulation
- app CPU viewer integration for native `float32` layers is implemented with exact f32 pick readout
- app full time-series and ROI-local analysis integration for native `float32` layers is implemented through exact CPU source-value reads
- exact `float32` brick reads, CPU resident-brick `float32` rendering, renderer-level GPU bricked atlas `float32` rendering, and app resident-streaming integration are implemented
- `float32` app resident-streaming uses typed f32 worker/cache/resident payloads rather than display-compatible `uint16` bricks
- `float32` GPU analysis acceleration is implemented for full-volume and ROI-local summaries through explicit f32 partial-reduction provenance; CPU remains the exact `f64` accumulation reference path
- native dense intensity writers emit a required brick range hierarchy for valid-data presence, valid voxel counts, min, and max
- validation recomputes the range hierarchy from brick records and rejects missing or inconsistent acceleration metadata
- the data engine exposes per-brick metadata without decoding payloads, and the app resident-streaming scheduler uses `occupied=false` only for no-valid-data bricks while treating unmasked zero-valued bricks as valid data

Percentile defaults:

- `0.1`
- `1.0`
- `50.0`
- `99.0`
- `99.9`

Histograms and percentiles are display/diagnostic metadata. They must not alter stored scientific values.

Dense intensity brick occupancy is validity-derived metadata:

- `occupied = true` when the brick contains at least one valid voxel
- `occupied = false` when every voxel in the brick is render-invalid
- layers without no-data masking treat every geometrically present voxel as valid, including zero-valued voxels
- all bricks still record min/max metadata
- in the current sharded v1 writer, all logical bricks are represented inside
  sharded storage and every shard payload is checksummed, so full validation can
  verify encoded payload integrity without treating missing fill-value chunks as
  valid data

For layers declaring no-data masking:

- `occupied = true` when the brick contains at least one render-valid voxel
- `occupied = false` when every voxel is render-invalid after the one-voxel invalid dilation policy
- brick min/max, histograms, percentiles, and range hierarchy use render-valid voxels only
- any omitted or special-cased all-invalid payload must be described explicitly in metadata and remain strictly validateable

The current range hierarchy is derived from leaf brick records. Level `0` mirrors the brick grid, and each higher level groups spatial bricks by `2 x 2 x 2` independently for each timepoint. Each range record stores whether any child has valid voxels, the summed valid voxel count, and the valid-value min/max interval.

## Checksum And Validation Policy

Checksum algorithm:

- `blake3`

Defaults:

- production sharded arrays: checksum each shard payload
- explicitly scoped unsharded test fixtures: checksum each chunk payload
- manifest records checksum algorithm and expected digest location

## Checksum Digest Mapping

Checksums apply to encoded storage payloads, not decoded scientific values.

The pre-refactor unsharded v1 brick/chunk record also stores `payload_bytes`,
the exact encoded Zarr chunk object byte length for that logical coordinate.
After the sharded production hard cutover, production encoded payload byte
counts live in shard-level records.

Digest records must declare:

- `algorithm`
- `scope`
- `array_path`
- logical payload coordinate
- hex digest

Accepted checksum scopes:

- `zarr_chunk_payload`: digest of one encoded Zarr chunk object in an unsharded array
- `zarr_shard_payload`: digest of one encoded Zarr shard object in a sharded array

Rules:

- For explicitly scoped unsharded test fixtures, brick/chunk records may carry
  `zarr_chunk_payload` checksums indexed by logical chunk coordinate.
- For sharded production arrays, per-brick `payload_checksum` is not used by
  default; shard digest records live in shard-level metadata indexed by logical
  shard coordinate.
- For compressed unsharded test-fixture chunks, the digest is over the encoded
  compressed chunk payload after Zarr codecs, before decoding.
- For sharded arrays, the digest is over the encoded shard object. Per-chunk corruption inside a shard is reported as a shard validation failure unless a future schema adds optional per-chunk logical checksums.
- Quick validation checks that required digest records exist and match expected coordinates.
- Full validation resolves payloads through the Zarr store layer and verifies digests before or during decode.
- App and renderer code must not infer digest locations from file names or Zarr internal paths.

Validation modes:

- quick validation: manifest, schema, metadata consistency, paths, dimensions, dtype, transforms, and required index presence
- full validation: quick validation plus checksum verification and payload decode/read tests

The viewer open path should run quick validation before entering the viewer. Full validation is a user/developer action or preprocessing completion gate.

## OME-Zarr Metadata Policy

Mirante4D may mirror useful OME-NGFF metadata for interoperability, especially axes, multiscales, coordinate transforms, channels, and labels.

Mirante4D metadata is authoritative for the app.

Rules:

- no Mirante4D-required field may exist only in mirrored OME metadata
- no generic OME-Zarr dataset is accepted without `mirante4d.json`
- mirrored metadata must not contradict authoritative Mirante4D metadata
- contradictions are validation errors

## Invariants

- Channels are separate layers.
- Base scale preserves the source-like grid.
- Time is not downsampled by default.
- Required acceleration metadata is not optional for renderable production datasets.
- Compression never implies lossy value conversion.
- Chunk, shard, reduction, and checksum policies are explicit in metadata.
- Unsupported codecs fail clearly.

## Failure Modes

- logical brick, subchunk, or shard shape missing or inconsistent with Zarr
  metadata
- shard metadata missing payloads
- unsupported compression codec
- multiscale shape or transform mismatch
- missing histogram or brick min/max for renderable scale
- checksum mismatch
- mirrored OME metadata contradiction

## Testing Requirements

- brick-shape and shard-layout validation tests
- zstd encode/decode and unsupported-codec rejection tests
- multiscale shape/transform tests
- histogram and percentile deterministic tests
- per-brick min/max/occupancy tests
- range hierarchy consistency tests
- checksum quick/full validation tests
- OME metadata contradiction rejection tests

## Open Questions

- Whether byte shuffling becomes default after benchmarks.
- Whether an additional max-pyramid is worth storing for MIP-heavy workflows.
- Exact external index format for very large brick metadata.
